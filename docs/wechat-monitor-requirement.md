## 一、业务背景

### 1.1 业务场景

小九九公司主营跨境电商服务，通过微信群与客户沟通：
- 客户咨询店铺购买、代入驻等服务
- 30% 订单来自老客户（朋友圈转化）
- 70% 订单来自新客户（小红书引流）

### 1.2 当前痛点

1. **人工监控效率低**：需要实时盯着微信群，容易遗漏重要消息
2. **记录不规范**：客户意向记录在个人笔记，难以统一管理
3. **响应不及时**：错过客户咨询的黄金时间，影响转化率
4. **数据难分析**：无法统计客户意向、热门需求等数据

### 1.3 业务目标

1. **自动化监控**：实时监控微信群消息，识别客户意向
2. **结构化记录**：自动记录到飞书多维表格，便于管理和分析
3. **及时响应**：5-10 分钟内发现重要消息，提升响应速度
4. **数据驱动**：积累客户意向数据，优化营销策略

## 二、需求约束

### 2.1 硬性约束

| 约束项 | 要求 | 说明 |
|--------|------|------|
| **企业资质** | ❌ 无企业微信、无公众号 | 仅有个人微信账号 |
| **风险承受** | ❌ 不能接受封号风险 | 必须零风险方案 |
| **监控频率** | ✅ 5-10 分钟轮询 | 平衡效率和风险 |
| **功能需求** | ✅ 监控 + 记录 | 无需自动回复 |
| **部署环境** | ✅ Mac + Docker | 现有 OpenClaw 环境 |

### 2.2 技术约束

1. **不能修改微信客户端**：避免触发风控
2. **不能使用微信协议**：Web 版已停止服务
3. **必须符合 OpenClaw 架构**：通过 MCP 插件集成
4. **必须复用现有能力**：Haystack-API + GLM-4.6V

## 三、技术方案

### 3.1 方案选型

经过调研，确定采用 **RPA + 视觉识别** 方案：

| 方案 | 优势 | 劣势 | 风险 | 推荐度 |
|------|------|------|------|--------|
| **RPA + 视觉识别** | 完全非侵入、零封号风险 | 需要 Mac 保持微信打开 | 🟢 低 | ⭐⭐⭐⭐⭐ |
| 官方 WeChat API | 零风险、官方支持 | 需要企业资质 | 🟢 低 | ⭐⭐⭐⭐ |
| Wechaty SDK | 开源、社区支持 | 存在封号风险 | 🟡 中 | ⭐⭐ |
| 浏览器自动化 | 技术成熟 | Web 版已停止 | 🔴 高 | ⭐ |

### 3.3 核心组件

#### 3.3.1 RPA 层（AppleScript）

**职责**：
- 定位微信窗口
- 截取聊天区域（最新 N 条消息）
- 保存截图到临时目录

**技术选型**：
- **AppleScript**：Mac 原生自动化
- **macOS Accessibility API**：读取窗口元素
- **screencapture**：系统截图命令

**关键代码示例**：
```applescript
tell application "WeChat"
    activate
    delay 0.5
end tell

tell application "System Events"
    tell process "WeChat"
        -- 获取聊天窗口位置
        set chatWindow to window 1
        set {x, y} to position of chatWindow
        set {w, h} to size of chatWindow

        -- 截取聊天区域
        do shell script "screencapture -R" & x & "," & y & "," & w & "," & h & " /tmp/wechat_chat.png"
    end tell
end tell
```

#### 3.3.2 视觉识别层（GLM-4.6V）

**职责**：
- 接收截图
- 识别聊天消息结构
- 提取群名、发言人、消息内容
- 返回结构化 JSON

**技术选型**：
- **Haystack-API**：已有的视觉模型服务
- **GLM-4.6V**：支持图片识别的多模态大模型

**Prompt 设计**：
```
你是微信聊天记录识别专家。请从截图中提取最新的聊天消息。

**输出格式（JSON）**：
{
  "group_name": "群名称",
  "messages": [
    {
      "sender": "发言人昵称",
      "content": "消息内容",
      "timestamp": "时间（如：14:23）"
    }
  ]
}

**识别规则**：
1. 只提取最新的 5 条消息
2. 忽略系统消息（如"XXX 加入群聊"）
3. 如果无法识别，返回空数组
4. 保持原始文字，不要修改或总结
```

#### 3.3.3 MCP 服务器（Python）

**职责**：
- 定时触发 RPA 截图
- 调用 Haystack-API 识别
- 去重（避免重复记录）
- 暴露 MCP 工具接口

**核心接口**：
```python
# MCP 工具接口
@mcp_tool
def wechat_get_messages(group_name: str = None) -> List[Dict]:
    """
    获取微信群消息

    Args:
        group_name: 群名称（可选，不指定则返回所有群）

    Returns:
        [
            {
                "group_name": "跨境电商交流群",
                "sender": "张三",
                "content": "想了解法国店铺的价格",
                "timestamp": "14:23",
                "message_id": "unique_hash"
            }
        ]
    """
    pass
```

**去重策略**：
- 使用消息内容 + 时间戳生成唯一 hash
- 缓存最近 1 小时的消息 ID
- 避免重复记录同一条消息

#### 3.3.4 小冠 Agent（OpenClaw）

**职责**：
- 定时调用 `wechat_get_messages`
- 过滤关键词（店铺、入驻、购买、咨询、价格等）
- 记录到飞书多维表格
- 更新记忆（客户意向历史）

**Heartbeat 配置**：
```json
{
  "heartbeat": {
    "every": "10m",
    "prompt": "检查微信群消息，记录客户意向到飞书。"
  }
}
```

**关键词列表**：
```python
KEYWORDS = [
    # 购买意向
    "想买", "购买", "多少钱", "价格", "报价",
    # 店铺相关

    # 咨询相关
    "咨询", "了解", "问一下", "请问",
    # 紧急词汇
    "急", "尽快", "今天", "马上"
]
```

## 四、数据流设计

### 4.1 消息流转

```
1. RPA 截图（每 10 分钟）
   ↓
2. GLM-4.6V 识别
   ↓
3. 返回结构化数据
   {
     "group_name": "跨境电商交流群",
     "messages": [
       {
         "sender": "张三",
         "content": "想了解法国店铺的价格",
         "timestamp": "14:23"
       }
     ]
   }
   ↓
4. 小冠过滤关键词
   - 匹配到："法国店铺"、"价格"
   - 判断为：客户意向
   ↓
5. 记录到飞书多维表格
   {
     "来源": "微信群",
     "群名": "跨境电商交流群",
     "客户昵称": "张三",
     "消息内容": "想了解法国店铺的价格",
     "关键词": ["法国店铺", "价格"],
     "记录时间": "2026-03-13 14:25:00",
     "状态": "待跟进"
   }
   ↓
6. 更新记忆
   - 记录客户意向历史
   - 用于后续分析和推荐
```

### 4.2 飞书表格字段设计

**表名**：客户意向表（tblXXXXXXXX）

| 字段名 | 类型 | 说明 | 示例 |
|--------|------|------|------|
| 来源 | 单选 | 消息来源 | 微信群 |
| 群名 | 文本 | 微信群名称 | 跨境电商交流群 |
| 客户昵称 | 文本 | 发言人昵称 | 张三 |
| 消息内容 | 多行文本 | 原始消息 | 想了解法国店铺的价格 |
| 关键词 | 多选 | 匹配的关键词 | 法国店铺, 价格 |
| 意向等级 | 单选 | 高/中/低 | 高（自动判断） |
| 记录时间 | 日期时间 | 自动记录 | 2026-03-13 14:25:00 |
| 状态 | 单选 | 待跟进/已联系/已成交/已放弃 | 待跟进 |
| 跟进人 | 人员 | 负责人 | 小冠 |
| 备注 | 多行文本 | 跟进记录 | - |

## 六、风险评估

### 6.1 技术风险

| 风险项 | 风险等级 | 影响 | 缓解措施 |
|--------|---------|------|---------|
| GLM-4.6V 识别准确率不足 | 🟡 中 | 误识别或漏识别消息 | 优化 Prompt，增加后处理逻辑 |
| Mac 微信窗口变化 | 🟡 中 | 截图失败 | 增加窗口检测和自适应逻辑 |
| Haystack-API 不稳定 | 🟡 中 | 服务中断 | 增加重试机制，考虑本地 OCR 备份 |
| 消息去重失效 | 🟢 低 | 重复记录 | 完善 hash 算法，增加时间窗口 |

### 6.2 业务风险

| 风险项 | 风险等级 | 影响 | 缓解措施 |
|--------|---------|------|---------|
| 关键词过滤不准确 | 🟡 中 | 遗漏重要客户 | 持续优化关键词列表，增加人工审核 |
| Mac 需要保持开机 | 🟡 中 | 监控中断 | 配置 Mac 不休眠，增加监控告警 |
| 微信客户端更新 | 🟢 低 | RPA 失效 | 定期测试，快速适配新版本 |

### 6.3 合规风险

| 风险项 | 风险等级 | 影响 | 缓解措施 |
|--------|---------|------|---------|
