# Yunyi Tokens Usage Skill

按以下流程执行，不得跳步。

## 0) 安全边界

- API Key 仅用于网页登录，不写入 memory 文件。
- 回复中不得泄露完整 API Key，只能使用掩码（例如 `B370****K988`）。
- 不得把 API Key 输出到工具日志解释文本。

## 1) 前置判定（必须先做）

1. 调用 `session_status`。
2. 从返回文本中判断当前模型标识是否包含 `gpt`（不区分大小写）。
3. 若不包含，立刻返回：

`前置条件不满足：当前主模型非 GPT，未执行云驿 tokens 查询。`

并停止后续动作。

## 2) 读取 API Key（必须先读，不得在回复中泄露）

1. 调用 `read` 读取本地文件：

`/Users/xiaojiujiu2/.openclaw/workspace/xiaodong/skills/yunyi-tokens-usage/API_KEY.txt`

2. 将读到的内容作为 `apiKey`（去除首尾空白/换行）。
3. 若文件不存在/为空，直接返回失败：

`失败（原因: 缺少 API Key 文件: skills/yunyi-tokens-usage/API_KEY.txt）`

备注：新的 key 需要用户外部购买后提供；本技能不尝试自动生成/刷新 key。

## 3) 页面访问与登录

1. 默认调用 `browser`，使用 `target="host"`、`profile="chrome-relay"`。
2. 仅当用户明确要求接管整个已登录浏览器会话，或 relay 模式连续失败且用户在线可批准连接时，才允许改用 `target="host"`、`profile="user"`。
3. 默认复用用户当前已登录的 Chrome 当前标签页；开始前必须确保目标标签页已手动点亮 OpenClaw Browser Relay 扩展为 `ON`。
4. 打开 `https://yunyi.rdzhvip.com/user`。
5. 立刻 `snapshot` 确认页面状态，但必须优先使用尽量小的页面树：优先 `refs="aria"` 且只保留完成登录与读取四张卡片所需的最小深度/最小节点范围；禁止为了稳妥反复抓整页大快照。
6. 若检测到登录表单（出现 API Key 输入框与“开始使用”按钮）：
   - 向第一个 API Key 输入框输入 `apiKey`。
   - 点击“开始使用”。
   - 等待页面完成跳转后再次 `snapshot`，仍然使用最小必要范围。

禁止猜测任何备用域名。若 host 浏览器不可用，直接返回失败并说明 Browser Relay 未连接或网关浏览器当前不可用。若 `profile="chrome-relay"` 在线但拿不到页面或 `tabs=0`，优先返回“当前 Chrome 标签页未成功 Attach Relay”，提示用户在目标标签页重新点亮扩展到 `ON` 后重试；只有在上面的升级条件成立时才允许切到 `profile="user"`，不要擅自切回 `openclaw`。

## 4) 关键状态判定（区分: 页面问题 vs key 失效）

在每次 `snapshot` 后，按以下优先级判定。除非页面证据不足，否则不要继续扩大 snapshot 范围：

1. 站点/网络不可用：
   - 出现浏览器错误页（例如 5xx/522/ERR_*）或页面无任何可读文本但持续 loading。
   - 处理：进入重试策略（见第 6 节）；重试后仍失败则返回 `失败（原因: 页面不可达/渲染失败）`。

2. key 无效/失效（必须有明确证据才可归因）：
   - 仍停留在登录表单，且出现明确错误提示/Toast（例如“无效/错误/过期/失效/Unauthorized/Invalid key”等语义）。
   - 或者重复登录 2 次后仍回到登录表单，并且页面出现上述错误语义。
   - 处理：返回 `失败（原因: API Key 失效或不正确）`，并附“需要你提供新的 key，我会写入 API_KEY.txt 生效”。

3. 登录成功：
   - 能读到 4 张数据卡片的标题或关键字段（见第 5 节）。

注意：只要页面处于“白屏/转圈/资源加载中”，不得把问题归因到 key。

## 5) 核心数据读取（必须可复核）

目标：从页面顶部 4 张卡片读取并结构化整理字段（标题文案可能微调，但语义固定）：

- 每日额度（金额、使用百分比、刷新倒计时/重置时间）
- 总 TOKENS（总量、入、出）
- 请求次数（总请求、今日请求）
- 累计消费（累计金额、今日金额）

提取要求：

- 优先按“卡片标题文本”定位卡片并读取其数值，而不是依赖卡片顺序/布局。
- 优先提取四张核心卡片及其紧邻字段，不要把整页趋势图、模型消费明细、长列表明细一起带入上下文。
- 对数值做最小清洗：去掉逗号分隔符、统一单位（TOKENS/次/元），保留原始文本备查。
- 一致性校验：若能同时拿到入/出/总量，检查 `总量 ≈ 入 + 出`；不成立则标记 `疑似解析异常`。

如果只拿到部分字段，必须标注缺失项，不得猜测。

## 6) 异常重试策略（强化 loading/白屏场景）

当出现以下情况时触发重试：

- 页面资源报错（例如 522）
- 页面持续 loading/白屏（snapshot 读不到关键文本/卡片标题）
- 卡片区域未渲染

重试规则：

- 最多重试 2 次（总共 3 轮）
- 退避等待：第 1 次 2 秒，第 2 次 4 秒
- 每轮都重新 `open -> 最小 snapshot ->（必要时登录）-> 最小 snapshot -> 状态判定 -> 读取`

超过重试上限后返回失败原因，禁止编造数据。
注意：失败时不得向用户询问“面板 URL”，因为目标 URL 已固定。

## 7) 固定输出模板（卡片式，优先可读性）

按以下结构回复（尽量每行 1 个信息块；金额/数值保留原始文本，不要过度重算）：

- `YYYY-MM-DD HH:mm (UTC) | <状态> | 剩余 <剩余时间> | 到期 <过期时间>`
- `额度 $<剩余> / $<总额度>（<已用百分比> 已用，<刷新倒计时> 后刷新）`
- `TOKENS <总量>（入 <入>｜出 <出>）`
- `请求 <总请求>（今 <今日请求>）`
- `消费 $<累计>（今 $<今日>）`
- `来源 https://yunyi.rdzhvip.com/user`

状态填写规则：

- 成功：能读取 4 张卡片关键字段
- 部分成功：缺失任意块，输出 `部分成功（缺失: ...）`
- 失败：输出 `失败（原因: ...）`

告警规则（可选追加在状态后面，用于醒目）：

- 若剩余时间 < 6 小时：追加 `告警: 快到期`
- 若已用百分比 >= 80%：追加 `告警: 快用完`

如果失败，需写明卡在哪一步（前置判定/API Key 读取/登录/页面渲染/字段解析/一致性校验）。若出现 provider 或会话处理错误，优先缩小 snapshot 范围后重试，不要在同一轮继续扩大整页树。
